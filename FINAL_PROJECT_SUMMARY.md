c# BRSR-GRI Ontology-Guided Semantic Mapping Engine
**Comprehensive Project Summary**

This document serves as the master project summary, detailing the architecture, the 5 core phases, the semantic relationship definitions, empirical proofs of the outcomes, and sample artifacts generated by the ESG Ontology Mapping Pipeline.

---

## 🏛️ System Architecture

The project is designed as an end-to-end, multi-stage pipeline that extracts unstructured ESG (Environmental, Social, and Governance) data from PDF reports, structures it into an RDF Knowledge Graph, and employs a hybrid deterministic/LLM-based semantic engine to automatically map concepts between different reporting frameworks (BRSR and GRI).

```text
Phase 1: PDF Parsing          → Structured JSON (hierarchy-preserving)
Phase 2: Ontology Construction → RDF/OWL + Neo4j (BRSR + GRI ontologies)
Phase 3: Semantic Mapping      → Multi-evidence candidate mapping
Phase 4: Evaluation            → Confidence scoring + explainability
Phase 5: Agentic Workflow      → LangGraph orchestration (7 agents)
```

---

## 🚀 The 5 Phases & Outcomes

### Phase 1: PDF Parsing & Structured JSON
**Objective:** Parse raw BRSR and GRI PDF documents, segmenting their hierarchical structure (Sections → Disclosures → Metrics) and enriching them via NLP.
- **Details:** Uses a `SectionSegmenter` (for BRSR) and a `GRISegmenter` to construct hierarchical trees. Strict environmental filtering guarantees only relevant disclosures (like GRI 300 series and BRSR Principle 6) are processed to prevent noise.
- **Outcome:** Generates deeply structured JSON files representing the exact taxonomy of the source documents.

### Phase 2: Ontology Construction
**Objective:** Convert the structured JSON trees into a formalized RDF Knowledge Graph and a Neo4j-compatible format.
- **Details:** The `OntologyBuilder` leverages `rdflib` to map concepts onto custom Namespaces (e.g., `RSO.Requirement`, `RSO.Disclosure`). It maps definitions, datatypes, and hierarchical paths.
- **Outcome:** Exports `esg_ontology.ttl` (Turtle format) containing the semantic triples, alongside Neo4j CSVs and a Markdown Mermaid diagram of the ontology relationships.
- **Proof:** Successfully generated an ontology comprising **73 BRSR concepts** and **458 GRI concepts**.

### Phase 3: Semantic Mapping Engine
**Objective:** Autonomously discover mapping candidates between BRSR and GRI requirements using a multi-weighted similarity scoring algorithm, validated by a robust LLM.
- **Details:** 
  1. **Pre-filtering & Embeddings:** Embeds text using `sentence-transformers/all-mpnet-base-v2` and caches them to disk (`embeddings_cache.pkl`) for extreme speed.
  2. **Multi-Factor Scoring:** Calculates a final similarity score based on embeddings, lexical overlap, datatype/unit compatibility, and hierarchical depth (Graph Topological Similarity).
  3. **Batched LLM Verification:** Candidates scoring above `0.30` are dynamically batched (chunks of 10) and sent to an OpenRouter LLM (`meta-llama/llama-3.3-70b-instruct`). 
- **Outcome:** A serialized `mapping_repository.json` classifying overlaps.

### Phase 4: Evaluation & Explainability
**Objective:** Analyze the outputs from Phase 3 and present a human-readable audit dashboard.
- **Details:** Parses the `mapping_repository.json` and evaluates the LLM's classification vs. the Engine's mathematical baseline.
- **Outcome:** A highly interactive HTML Dashboard (`mapping_dashboard.html`) and a CLI statistical report.

## 🔄 Pipeline Linkage & Handoff

The pipeline phases are inherently decoupled but sequentially connected through highly structured intermediary artifacts. The most critical handoff occurs between **Phase 2 (Ontology Construction)** and **Phase 3 (Semantic Mapping Engine)**:

1. **Phase 2 Export:** At the conclusion of Phase 2, the `OntologyBuilder` serializes the constructed RDF graph into a localized file: `data/processed/ontology/esg_ontology.ttl` (Turtle format).
2. **Phase 3 Ingestion:** Phase 3 requires this `.ttl` file to initiate. The `SemanticMappingEngine` calls `self.graph.parse(ontology_path, format="turtle")`, completely reloading the Phase 2 knowledge graph back into memory. This graph structure provides the foundational topology needed to calculate "Graph Topological Similarity" and extract parent-child context before prompting the LLM.

## 🔗 Semantic Relationships

The Semantic Mapping Engine evaluates concepts and assigns them standard SKOS-inspired alignment relationships based on the calculated similarity score and LLM verification:

* **Equivalent:** Concepts represent exactly the same disclosure requirement.
* **Partial Equivalent:** Concepts share a meaningful conceptual overlap, even if one framework requires slightly more granularity than the other (e.g., BRSR Scope 1 Emissions vs GRI 102-5).
* **NotMapped:** Concepts are deemed unrelated or the LLM verified that they cover fundamentally different ESG topics (e.g., Water vs Emissions).
