================================================================================
BRSR-GRI SEMANTIC MAPPING FRAMEWORK: CURRENT FLAWS & FUTURE SUGGESTIONS
================================================================================

This document reviews the architectural and logic limitations (flaws) of the 
current mapping framework and outlines actionable suggestions for optimization.

--------------------------------------------------------------------------------
1. CURRENT FLAWS & LIMITATIONS
--------------------------------------------------------------------------------

FLAW 1: REASONING WITHOUT GROUND TRUTH (GT)
* Description: The pipeline treats Groq's decisions (llama-3.3-70b-versatile) as 
  the absolute "Ground Truth" (GT) to measure accuracy, precision, and recall.
* Impact: Groq is still prone to hallucinations or simple semantic mistakes.
  Measuring other advanced models (like OpenAI or Claude) against Groq creates a
  biased benchmark that doesn't reflect actual performance.

FLAW 2: SEVERE SYNTACTIC DEPENDENCY IN VECTOR RAG
* Description: Phase 3 selects GRI candidates by comparing disclosure labels
  using simple TF-IDF and dense embeddings.
* Impact: Concepts that are semantically identical but use different vocabulary
  (e.g., "greenhouse gas emissions" vs. "Scope 1 emissions intensity") may get
  low similarity scores and be excluded from candidates entirely.

FLAW 3: NO EXPLAINABILITY OR CHAIN-OF-THOUGHT (CoT)
* Description: LLMs are forced to output a JSON array of "Agree" or "Disagree" 
  directly without any explanation.
* Impact: Forcing a model to output raw tokens without letting it write its 
  reasoning down first (Chain of Thought) lowers reasoning accuracy and makes 
  verifications impossible to audit.

FLAW 4: DISK IO BOTTLENECK IN REAL-TIME CHECKPOINTING
* Description: Every successful thread batch triggers a full read, modify, and
  write of the entire multi_llm_results.json and evaluation_cache.json.
* Impact: As mapping counts grow into the thousands, concurrent threads will
  experience significant file locking and IO blockages, degrading performance.

FLAW 5: STATIC OFFLINE CSV EXPORTS
* Description: Phase 2 exports Neo4j CSV files but does not interact with a live
  graph database.
* Impact: The mapping engine cannot query graph relationships (e.g., parent/child
  node paths) in real time to refine candidates.

--------------------------------------------------------------------------------
2. SUGGESTIONS & OPTIMIZATIONS
--------------------------------------------------------------------------------

SUGGESTION 1: HUMAN-IN-THE-LOOP GOLD STANDARD
* Action: Curate a human-verified dataset of 50-100 BRSR-GRI mapping pairs.
* Benefit: Establishes a true, unbiased ground truth to benchmark LLMs accurately.

SUGGESTION 2: IMPLEMENT GRAPH-RAG (NEO4J ACTIVE QUERIES)
* Action: Connect the Phase 3 mapping engine directly to a running Neo4j instance.
* Benefit: Allows candidates to be selected using structural graph traversals
  (e.g. searching for sibling/parent topics) rather than simple text similarity.

SUGGESTION 3: ENHANCE VERIFIER WITH EXPLAINABLE REASONING
* Action: Update the LLM prompt template to accept structured explanations:
  Format: {"id": 0, "reasoning": "...", "decision": "Agree"}
* Benefit: Leverages Chain-of-Thought (CoT) to raise decision accuracy and provides
  a clear audit trail.

SUGGESTION 4: MIGRATE CACHE & CHECKPOINTS TO SQLITE
* Action: Replace JSON files with an embedded SQLite database.
* Benefit: Native database indexing eliminates file locking bottlenecks, making
  multi-threaded writes fast and safe.

SUGGESTION 5: DYNAMIC TEMPERATURES & ENSEMBLE CONSENSUS
* Action: Let the verifier query multiple models simultaneously and use a voting
  consensus (majority wins) with a slight temperature (> 0.2) to find matches.
* Benefit: Reduces reliance on any single LLM endpoint and yields higher quality,
  resilient mapping selections.
